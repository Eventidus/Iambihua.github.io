Docker介绍

  1. 虚拟化技术与容器技术
    1. 虚拟化技术的理解
      1. 它是一种将计算机物理资源抽象化，转化成计算机虚拟资源以提供给程序使用
    2. 虚拟化技术很好地解决了跨平台兼容的问题
    3. 虚拟化只是提高了计算机计算机资源的利用率，并不会减少资源的占用。
    4. 资源隔离
      1. 将程序分隔在不同的运行环境中
    5. 容器技术
      1. 操作系统提供接口，让多个应用程序在不同的环境中独立运行，互不干扰。
  2. Docker介绍
    1. Docker属于操作系统层面的虚拟化技术
    2. Docker与传统的虚拟化技术的不同之处
      1. Docker没有虚拟出硬件，并且它的容器直接运行于宿主内核，容器内并没有自己的内核
      2. 传统的虚拟化技术需要虚拟出硬件，在其上运行一个操作系统，并且在这个操作系统里面运行应用程序
    3. 命名空间
      1. 集合相同模块的类，区分不同模块的同名类
  3. Docker镜像
    1. 是一种特殊的文件系统，提供了 容器运行需要的库，配置文件，等等，还包含了容器运行需要的配置参数，它的核心精髓是保证了环境的一致性
    2. Docker镜像技术是一套统一的接口技术，它适配了不同的开发语言
      1. Docker镜像采用分层存储结构，每一次的修改都只对当前所在的一层有效，对前面几层不起作用
    3. 中间层镜像:
      1. 是其他镜像依赖需要的镜像
      2. 用于提高镜像复用性，加速镜像构建
    4. 删除镜像
      1. 两类删除行为：Untagged和Deleted
      2. 有时候删除镜像，可能仅仅是取消了镜像的标签，也就是Untagged的意义，真正影响镜像删除的，要考虑其他镜像是否依赖当前要删除镜像的某一层，或者有容器依赖当前要删除镜像的某一层，或者有 别个标签指向了当前要删除的镜像，这种情况下就不会触发镜像删除行为，也就是Delected行为
  4. Docker容器
    1. 容器是基于镜像构建，是独立运行的一个或一组应用
    2. 容器也是分层存储的，容器在作为基础层的镜像之上又构建用于容器运行的存储层
    3. 启动容器时，如果需要的镜像不存在，会自动pull下载
    4. 容器里只会存在必须的资源，多余的资源不会存在，所以Docker对资源的利用率很高
    5. 一个容器只有在执行任务时才会处于running状态，不管是否启动
    6. 启动容器的命令为docker run <image>，如果要指定容器在后台运行可以指定-d参数
    7. 启动处于exited状态的容器 docker container start [container-id]
    8. 列出正在执行任务也即running状态的容器 docker container ls
    9. 列出容器运行结果 docker container logs [containerId]
    10. 对处于后台运行的容器进行操作可以使用docker exec或者attach命令，但推荐使用exec命令，因为exit容器后，不会终止容器
    11. docker container rm只能删除处于exited的容器，如果要删除处于running状态的容器,可以加-f参数
    12. 删除所有处于exited状态的容器 docker container prune
    13. docker diff [container]查看对容器的更改
    14. docker commit -a xxx -m xxx containerName [REPOSITORY[:TAG]] 从容器创建一个新镜像
    15. docker history container查看镜像的历史记录
  5. Docker数据卷
    1. 数据卷是容器用于管理数据的场所
    2. 数据卷的特性
      1. 在多个容器之间共享
      2. 对数据卷的生效会即时生效
      3. 删除容器后，数据卷并不会消失
      4. 对数据卷的更新，不会影响镜像
    3. 创建数据卷 docker volume create [volume-name]
    4. 查看某个数据卷的具体信息docker volume inspect [volume-name]
    5. 删除数据卷docker volume rm，如果要在删除容器的同时删除数据卷可以加上-v参数
    6. 删除未挂载到容器的数据卷docker volume prune
  6. Docker网络配置
    1. -p -P都能指定端口
      1. -P随机指定一个主机端口映射到容器开放的端口
      2. -p主动指定端口映射
        1. 几种格式的端口映射
          1. hostPort:containerPort
            1. 这种情况下，不管本地的ip地址是啥，只要主机端口相同，他们指向同一个地址，也就是映射所有接口地址
          2. ip:hostPort:containerPort
            1. 映射指定地址的映射端口，只能通过指定的ip地址加端口访问
          3. ip::containerPort
            1. 映射到指定地址的任意端口，也就是说对于指定的ip地址，随机指定一个主机端口
    2. Docker 容器互联
      1. 用自定义网络来互联容器
        1. docker network create -d bridge my-net 其中-d参数指定网络类型
        2. docker run –network my-net busybox 使用–network参数 指定busybox容器链接到mynet网络
    3. Dockerfile
      1. 是一个文本文件，以指令为基础一层层构建镜像
      2. 如果不以任何镜像为基础，可以使用scratch镜像，scratch镜像是一种虚拟空白镜像
      3. 使用指令书写命令的原则
        1. 能写在一层的命令尽量写在一个指令里，因为一个指令构建一层镜像
        2. 尽力确保镜像只包含自己需要的东西，多余的东西记得及时清理，因为当前一层的镜像只能在当前一层处理，在下一层处理无效，负责会随着镜像的构建存在，造成镜像臃肿
      4. docker build xxx命令用于构建镜像
      5. Dockerfile的文件名并不固定
      6. Dockerfile文件指令
        1. RUN指令用于运行命令，每执行一次就会就会狗创建一层镜像
        2. COPY指令用于将源文件复制到目标文件
        3. ADD指令相当于进化版的COPY指令，只是增加了一些功能，ADD指令更适用于自动加压缩，权限调整等场合
        4. CMD指令跟RUN指令类似，只不过RUN指令用在容器构建时，而CMD指令用在指令运行时，CMD指令为启动的容器指定默认要执行的程序，只能有一个，如果有多个，则最后一个覆盖前面的，但它会被docker run参数中指定的命令覆盖
        5. ENV指令 设置环境变量，这样可以在全局环境中通过$引用这个环境变量
        6. ARG指令与ENV指令环境变量一致，不过它的作用域在Dockerfile内
        7. Volume指令定义匿名卷，容器会自动挂载到匿名卷
        8. EXPOSE指令声明要暴露出去的指令
        9. WORKDIR指令指定容器工作的目录
        10. USER指令用于指定执行后续程序需要的用户和用户组
        11. HEALTHCHECK指令用于指定程序监听docker容器服务的运行状态
    4. Docker Compose
      1. 用于实现Docker容器集群的快速编排
      2. 本质是定义和运行有多个容器的应用
      3. 两个概念
        1. 服务
          1. 多个有相同镜像的实例
        2. 项目
          1. 相关联的多个容器组成的业务单元
      4. Compose用python语言编写，可通过pip安装
      5. Docker Compose使用YAML文件对容器进行管理
      6. Compose模板文件
        1. build指令指定Dockerfile文件所在的目录，可以利用这个指令构建镜像，而不用再从compose模板文件书写
          1. context指令指定dockerfile文件所在目录
          2. dockerfile指令指定dockerfile文件名
          3. args指令指定构建镜像时的变量
          4. cache-form指令指定构建镜像时的缓存
          5. cap-add指令指定为容器增加的内核能力
          6. cap-drop指令指定为容器去掉的内核能力
          7. command指令指定容器启动后默认执行的命令
          8. container_name指令指定容器名称
          9. devices指令设定设备映射关系
          10. depends_on指令设备的依赖，先后启动的问题
          11. dns指令定义DNS服务器
          12. dns_search指令用于配置DNS搜索域
          13. tmpfs指令用于指定挂载一个tmpfs文件系统到容器
          14. env_file指令用于指定环境变量的路径
          15. environment指令用来设定环境变量
          16. expose指令指定容器向外暴露的端口，但不映射到宿主机
          17. external_links指令指定连接的compose外部的容器
          18. image指令指定镜像
          19. labels指令为容器添加元素据信息
          20. network_mode指令用来设置网络模式
          21. ports指令指定主机以及容器端口
          22. secrets指令用于存储敏感数据

Swiper使用方法

  1. npm i swiper安装库
  2. 在需要的组件中引入swiper库
  3. 在入口文件引入swipe样式库
  4. 创建Swiper实例(在mounted方法中，但不可以直接书写，需要使用setTimeOut延迟创建swiper实例)
  5. 针对第四步，最优的方法是使用侦听器watch和nextTickhans函数
    1. 使用nextTickhans的原因是Swipper实例创建成功时，V-for循环未必执行完毕
  6. 如果不是在当前组件发起请求，可以直接在mounted方法中创建Swiper实例
  7. 不生效的原因可能是
    1. 在请求完成之前已经创建Swipper实例
    2. V-for循环未执行完毕
    3. 没有及时监听到数据变化，需要设置immediate属性

异步操作

  1. JavaScript是单线程模型，JavaScript脚本只在一个线程上运行，这个线程叫主线程。
  2. Web Worker标准，允许JavaScript脚本创建多个线程，但是子线程只受主线程影响，且不得操作DOM。
  3. 同步任务
    1. 在主线程上排队，只有上一个任务执行完成，才会执行下一个任务。
  4. 异步任务
    1. 在任务队列排队，不用管当前任务是否执行完毕，可以直接执行下一个任务。
  5. 任务队列：存放需要执行的所有异步任务。
  6. 异步操作的流程控制
    1. 串行执行，耗时
    2. 并行执行，省时但耗资源
    3. 串行并行结合，省时同时也比较节约资源。

this 关键字

  2. this 的值为属性或方法当前所在的对象(环境)。

  3. 使用场合 1. 在全局环境下，this 的值为顶层环境 window。 2. 在构造函数中，this 指向实例对象。 3. 在对象的方法中，this 指向当前方法所在的对象，但如果对一个对象进行赋值时，this 的指向就会改变了。

Promise 对象

  1. Promise 是一种异步编程的解决方案，简单来说就像一种容器，里面保存着某个只有未来才会结束的事件的结果。
  2. Promise 的三种状态，pending(进行中),fulfilled(已完成),rejected(已失败)。
  3. Promise 的最终状态由异步操作之后的结果决定，并且状态改变之后不会再变。
  4. Promise 内部的错误不会影响到 Promise 外部的代码

Module 语法

  1. ES6 模块可在编译时(即静态阶段)确定模块的编译关系，而 CommonJS 只能在运行时确定模块的依赖关系，所以 ES6 模块可实现静态优化。
  2. ES6 模块支持按需加载需要的模块方法，但 CommonJS 只能加载所有的模块方法。
  3. ES6 模块自动采用严格模式。
  4. export 命令规定模块的对外接口，import 命令用于从模块中导入需要的功能。
  5. export 命令可以输出变量，函数和类。
  6. 通过 import 命令导入的模块，可在模块导入语句的头部提前执行。

Module 的加载实现

  1. 在浏览器中加载 ES6 模块
    1. <javascript>标签的两种属性 async 与 defer 的相同点是都是异步加载。
    2. <javascript>标签的两种属性 async 与 defer 区别
      1. defer：页面渲染完后执行。
      2. async：脚本下载完成后执行，同时中断当前的渲染。
      3. 多个 defer 脚本按顺序执行，多个 async 脚本执行顺序不能保证。
    3. 模块顶层的 this 指向 undefined。
    4. 在浏览器中加载 ES6 模块时，需要在<javascript>标签中指定 type=”module”属性。
    5. 浏览器中的 ES6 模块，默认是 defer 属性的异步加载。

Generator 函数语法

  1. Generator 相当于一个状态机，里面封装了多种内部状态

  2. 创建 Generator 函数时，Function 关键字跟函数名之间要有*星号

  3. 调用 Generator 函数后，不会执行里面的代码，而是生成一个迭代器对象

  4. 返回的迭代器对象可以反复使用 next()方法迭代内部状态

  5. 使用 next 方法会返回一个对象，这个对象有两个属性 1. value：当前状态的值，即 yield 表达式的值 2. done：迭代是否结束
