CORS 通信

  1. CORS（Cross-Origin-Resource-Sharing），中文名跨域资源共享。
  2. CORS 的作用是允许浏览器向跨域服务器发送 XMLHttpRequest（XHR）。
  3. 实现 CORS 跨域通信，主要看服务器是否实现 CORS 接口，也就是说主要在服务器端配置
  4. 服务器端配置 CORS 跨域的方法是设置 Access-Control-Allow-Origin
  5. CORS 可分为两种请求：
    1. 简答请求
      1. 简答请求需要请求方法和头信息字段都满足条件
      2. 请求方法
        1. HEAD
        2. POST
        3. GET
      3. 头信息字段
        1. Accept
        2. Content-Language
        3. Accept-Language
        4. Last-Event-ID
        5. Content-Type：只限于三个值 application/x-www-form-urlencoded、multipart/form-data、text/plain
      4. 不满足上面两个条件的，称为非简单请求。
      5. CORS 通信的基本流程：
        1. 浏览器发起 CORS 请求，并且在头信息增加 origin 字段，表示要发起跨域请求的域。
        2. 服务器判断浏览器发过来的 origin 是否在许可范围内。
        3. 如不在许可范围内，则发送正常的 HTTP 回应，但不会包含 Access-Control-Allow-Origin 字段。
        4. 如在许可范围内，发送的 HTTP 回应会包含以下字段
          1. Access-Control-Allow-Origin 字段： 表示服务器可允许跨域的域名，如果是*，则表示可接受任意域名。
          2. Access-Control-Allow-Credentials：表示是否允许发送 cookie。
          3. Access-Control-Expose-Headers：表示是否返回其他非基本字段。
        5. 要想发送 cookie 给服务器，开发者需要将 xhr.withCredentials 设置为 true，注意有些浏览器的 xhr.withCredentials 默认为 false。
    2. 非简单请求
      1. 非简单请求在发送 CORS 请求前，需要先发出预检请求。
      2. 预检请求是浏览器询问是否接受当前网页的域名，以及可接受哪些 HTTP 方法和字段。
      3. 预检请求的请求方法是 OPTIONS。
      4. 预检请求的头信息包含的部分字段如下
        1. origin
        2. Access-Control-Request-Method，发送 CORS 请求用到的方法。
        3. Access-Control-Requset-Headers，发送 CORS 请求额外发送的头部信息
      5. 如通过预检请求，则每次的 CORS 请求就跟简单请求一样。
      6. CORS 和 JSONP 的比较
        1. JSONP 只支持 Get 请求，而 CORS 支持所有类型的请求。
        2. JSONP 可以在老式浏览器上运行，也可以向不支持 CORS 的网站上请求数据。
